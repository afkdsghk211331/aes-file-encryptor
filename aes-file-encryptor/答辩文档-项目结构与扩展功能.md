# 项目结构与扩展功能

---

这个项目的整体结构不算复杂，围绕一个核心源文件展开，加上构建脚本、TypeScript 配置、测试页面和文档，形成了一套完整的 npm 包工程。

源代码只有一个文件，就是 src/index.ts，所有加密和解密的逻辑全部集中在这里面。TypeScript 的配置开的是严格模式，编译目标是 ES2020，模块格式是 ESNext，同时生成类型声明文件。因为用了 bundler 模式的模块解析，所以源码里可以直接用 import 的方式引入以后的目标格式，不需要写扩展名。

构建脚本 build.mjs 是自己写的，而不是用 webpack 或者 rollup 之类的打包器。原因很简单——TypeScript 编译器本身就能输出 JavaScript，不需要额外的打包步骤。构建流程分两步走：第一步用 tsc 编译出 ESM 格式的 JS，然后把 .js 后缀改成 .mjs；第二步用 tsc 以 CommonJS 模块格式重新编译一次，输出到临时目录，把 JS 文件拷回 dist 目录并改成 .cjs 后缀。最终 dist 目录下有三个文件：index.mjs 给 ESM 消费者用的，index.cjs 给 CommonJS 的 require 用的，index.d.ts 给 TypeScript 项目提供类型信息。

package.json 里的 exports 字段针对 import 和 require 两种使用方式分别映射了不同的文件，同时也指定了 types 字段指向声明文件。这样不管是用户写 import 还是写 require，npm 都能给出正确的文件，编辑器也能拿到准确的类型提示。另外 files 字段只包含了 dist 目录，发布到 npm 的时候不会把源码、测试和构建脚本打包进去。

测试部分就是一个 HTML 页面 test/test.html，直接在浏览器里打开就能跑。它从 dist 目录导入构建好的 ESM 产物，覆盖了各种边界情况：空文件、单字节、15 字节——15 这个数字不是随便选的，因为 AES 的块大小是 16 字节，15 字节刚好是不足一个块、需要填充的最小情况；然后是 16 字节——刚好一个块、不需要填充；17 字节——刚好超过一个块一点。再往上还有 100 字节、1 KB、10 KB、1 MB、2 MB，逐步验证了不同规模下的正确性。另外还覆盖了全零数据、全 0xFF 数据、覆盖 0 到 255 的所有字节值，以及错误密钥和模式不匹配的拒绝测试。每个测试用例在 CBC、CTR、CFB 三种模式下各跑一遍，总共 45 个加密解密往返测试，加上两个错误场景验证，加起来 47 个断言。

---

在加密的基本操作之外，这个项目做了几个扩展性的设计，每一层都有自己的考量。

第一层是密钥派生。直接用用户输入的密码作为 AES 密钥是绝对不安全的，因为密码通常不够长也不够随机。所以这里用了 PBKDF2-SHA256，这是一个基于密码的密钥派生函数，它的核心思想是用一个叫"盐"的随机值跟密码混合，然后反复哈希 10 万次。10 万次迭代的意义在于拉大暴力破解的成本——攻击者试一次密码就要算 10 万次 SHA-256，而我们正常使用只在加密和解密时各算一次，几乎感觉不到延迟。盐是 16 字节的随机值，每次加密都会生成一个新的，存储在加密文件的头部。这保证了即使用同一个密码加密不同的文件，派生出来的密钥也是不同的。

第二层是分块处理。浏览器环境的内存是有限的，如果把一个几 GB 的文件整个读进内存再加密，页面直接就会崩溃。所以所有三种模式都采用了 1 MiB 的分块策略——每次只读取和处理 1 MB 的数据，处理完一块释放一块的内存。进度回调也是基于分块实现的，每处理完一块就报告一次进度，这样 UI 上可以画一个平滑的进度条，用户能看到文件处理到百分之几了。

CBC 模式在分块的基础上做了一层额外的设计——每个分块通过 HKDF 从主密钥派生出一个独立的子密钥。HKDF 是 HMAC-based Key Derivation Function，它可以从一个主密钥派生出多个彼此独立的子密钥。具体做法是把分块索引编码成两个字节作为 HKDF 的上下文信息，这样不同分块派生出的密钥在密码学上是相互独立的。再结合每个分块自己独立的随机 IV，就实现了分块间的完全隔离。这个设计打破了标准 CBC 必须串行解密的限制，允许对加密文件做随机访问——如果你想解密文件中间某一块，不需要从头开始，直接定位到目标分块、派生它的子密钥、读它的 IV，解密这一个块就够了。

第三层是完整性校验。加密能保证数据不被看懂，但不能保证数据没被改过。如果有人在传输过程中篡改了密文，或者用户输错了密码，CBC 和 CTR 在解密时是不会主动报错的——它们只会静默地输出乱码。所以需要额外的一层保护。这个项目用的是 HMAC-SHA256，它在所有加密数据生成完之后，对整个文件头加上密文计算一个 32 字节的签名，附在密文前面。解密的时候先验签名，如果发现签名不匹配，直接抛异常，不输出任何错误数据。签名的验证采用了恒定时间的字节比较——就是让比较循环不管在哪个位置发现不匹配都跑完全程——这样可以防止攻击者通过测量响应时间来猜出正确签名的前几个字节。HMAC 的密钥也是通过 HKDF 从主密钥派生出来的，只不过用了不同的上下文标记，确保它跟加密用的密钥在密码学上不相关。

第四层是自定义文件格式。很多加密库只输出原始密文，IV、盐这些参数让用户自己去管理，但这样很容易出错——忘了存 IV 文件就解不开了，IV 和密文对不上也会出问题。这个项目把模式标识、盐、IV、原始文件大小、HMAC 签名和密文全部打包在一个二进制文件里，解密的时候只需要文件本身加密码，不需要任何额外的参数。原始文件大小也存了 8 个字节，这是因为 CBC 有填充、CFB 分块时最后一块可能不满，解密出来的内容会比原始文件多一点尾巴，需要用这个长度来截断。

这四层设计——安全密钥派生、分块内存管理、完整性校验、自包含文件格式——构成了一个完整的前端加密解决方案。用户不需要理解 PBKDF2、HKDF、HMAC 这些密码学概念，他只需要输入一个文件和一个密码，剩下的全部由库内部处理。

## Web 可视化工具

除了加密库本身，该项目还包含一个配套的 Web 可视化工具，位于独立的 aes-file-encryptor-web 目录下，用于在浏览器中直接演示和使用加密功能。这个 Web 工具基于 React 18 构建，用 Vite 5 作为开发和打包工具，样式层用了 Tailwind CSS。

在开发阶段，Web 工具通过 Vite 的 resolve.alias 配置将 aes-file-encryptor 的导入路径直接映射到库的 TypeScript 源文件上，而不是引用 dist 目录中的构建产物。这样做的好处是开发时修改库代码能立即热更新到页面上，不需要手动重新构建库。当然生产构建时仍然使用正式的 npm 包依赖。

整个应用的核心逻辑都集中在 App.tsx 这一个文件里，它向下拆出了几个子组件。FileSelector 负责文件上传，支持拖拽和点击选择两种方式，拖入文件时边框会高亮反馈。ModeSelector 用三个并排按钮切换 CBC、CTR、CFB 模式，当前选中项高亮。ResultDisplay 在加解密完成后展示操作耗时、输出文件大小和一个下载按钮。BenchCard 用于展示三模式对比时每一个模式的单独结果卡片，同样包含耗时、大小和独立下载。

Web 工具支持明暗主题切换，提供了三个选项：Light、Auto 和 Dark。Auto 模式下会跟随操作系统的颜色方案偏好，并且监听系统偏好的实时变化——用户在系统设置里切换主题，页面会即时响应。Dark 模式通过 Tailwind 的 class 策略驱动，在 html 元素上挂载 dark class 来切换样式。为了消除页面加载时的主题闪烁，index.html 里有一段内联 script，在 React 挂载之前就从 localStorage 读取用户偏好或检测系统主题，提前给 html 加上 dark class。

界面的操作流程很直观。顶部可以切换加密还是解密模式，接着是文件上传区域，然后输入密码，再选择加密算法。点击 Encrypt 或 Decrypt 按钮后会执行单次操作，下方出现进度条，进度值从库的 onProgress 回调实时获取，以百分比显示。操作完成后展示结果信息和下载按钮——加密得到的文件名会自动加上 .encrypted 后缀，解密时则去掉这个后缀。

除了单次操作，Web 工具还提供了一个"Compare All 3 Modes"功能。这个按钮点击后会自动用同一个文件和同一个密码，依次执行 CBC、CTR、CFB 三种模式的加密，记录每种模式的耗时，然后把结果并排展示在三张卡片里。卡片下方还有一个速度对比的柱状图——用最慢的那个模式作为 100% 基准，其他模式按比例画出横条，一眼就能看出哪个模式最快、快多少。系统还会直接标注出最快的是哪个模式。

这个 Web 工具的意义在于，它把一个纯代码库变成了一个可以直接上手的可视化产品。你不需要写一行代码就能体验三种加密模式的实际效果和性能差异，对于演示、教学和快速验证都非常方便。
